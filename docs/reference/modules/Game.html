1. Architectural Purpose

The Game class establishes the central integration boundary encapsulating physical logic away from native visual outputs. Acting as the core orchestrator, it explicitly synchronizes frame drawing metrics while processing isolated mathematical model iterations. This architecture guarantees the internal data state remains cleanly decoupled and uncorrupted regardless of overlapping browser interfaces or runtime execution layers.

3. API Reference

3.1. addSerializable(serializable: Serializable): void

Registers a custom object or module for session persistence.

Parameters

Name

Type

Description

serializable

Serializable

The custom object to be saved in the active session.

Behavior

This method serves to add new custom serializables of the game being developed. This allows the game to save custom properties (like specific player power-ups or puzzle progress) in the browser session. These properties can be automatically restored if the user leaves and returns, continuing the game from the PLAYING state. Note that engine modules implementing Serializable are automatically detected and added.

3.3. update(deltaTime: number): void

Abstract method for processing game logic.

Parameters

Name

Type

Description

deltaTime

number

Time elapsed since the last tick (in milliseconds).

Behavior

Called by the engine on every game logic tick, but only when the game state is playing. The tick frequency is controlled by configs.game.tickInterval.

Example

update(deltaTime: number): void {
    const { grid } = this.modules;
    grid.shiftRowsDown(1);
}

3.4. render(): void

Abstract method for processing visual frames.

Behavior

Called by the engine on every render frame (based on browser refresh rate, typically 60fps). Use this for animations, particles, or visual effects that do not affect the logical game state.

Example

render(): void {
    // Logic for smooth animations
}

3.5. setupGame(): void

Abstract method for setting up the specific game instance.

Behavior

Called once after all engine modules are initialized and linked. Use this to set initial player state, subscribe to custom controls, or prepare level data.

Example

setupGame(): void {
    this.modules.text.setTextSize(FontSize.MEDIUM);
}

3.6. drawTitleScreen(): void

Abstract method for rendering the game’s welcome screen.

Behavior

Called when the engine is powered ON but the game has not yet been started. Executed during the main draw loop instead of update and render.

3.7. drawGameOverScreen(): void

Abstract method for rendering the screen after a game ends.

Behavior

Called when the game state is gameOver. Executed as the primary rendering logic during Game Over, replacing the standard render() call to allow for clean menus or high-score displays.

3.8. reset(): void

Resets the game to its initial state.

Behavior

This method performs a comprehensive reset of the game, including: - Clearing the Grid. - Resetting the Score and level. - Resetting the Time accumulators. - Clearing the active Session. - Restoring all custom game properties to their post-setup values via the InitialStateSnapshot.

This method is called by the GameEventRegistry during power cycles and manual restarts.

3.9. destroy(): void

Cleans up the game instance.

Behavior

Stops the P5.js loop, unbinds all control event listeners, and stops all playing sounds. Use this before disposing of a game instance to prevent memory leaks.

4. Session Management

The Game base class automatically orchestrates gameplay sessions by subscribing to system-level GameState events:

  • Restoring Sessions: When the state changes to PLAYING, the Game checks if there are any active sessions stored. If detected, it triggers a UI modal.

  • Loop Blocking: While the session restoration modal is open (session.isModalOpen()), the engine completely skips the draw() loop content (both logic and rendering) to prevent any background interaction while the user is choosing whether to resume. See Session for more details.

  • Clearing Sessions: The active session is automatically wiped when the state changes to GAME_OVER, ensuring that players do not resume a lost game.

5. Automatic State Restoration (Reset)

To provide a seamless developer experience, the Game class implements an internal mechanism to transparently cache and restore custom client properties.

When a game is instantiated and its setupGame() method is invoked, the base Game automatically captures a snapshot of the client’s custom properties. This ensures that any attributes added by the developer (e.g., player lives, custom counters, or game-specific flags) are saved in memory without requiring manual implementation.

When a RESET action is triggered, the Game class automatically restores these custom properties back to their original post-setup values. This eliminates the need for developers to write boilerplate code for resetting their own state variables. For technical details on how this is achieved, see the InitialStateSnapshot architecture.

6. Automated System Controls

The Game base class automatically registers listeners for system-level buttons:

  • POWER: Toggles engine state.

  • SOUND: Toggles master mute.

  • COLOR: Toggles theme color mode.

  • RESET: Resets the grid, score, clears the active session, and resets the game state.

  • START_PAUSE: Manages the game lifecycle:

    • Starts the game from the Title Screen.

    • Toggles pause during gameplay.

    • Restarts the game immediately if in a Game Over state (via resetGameOver), and clears the active session.

  • EXIT: Clears the active session and performs a hard stop of the draw loop. The specific behavior after stopping depends on the external orchestrator (e.g., returning to a menu).
